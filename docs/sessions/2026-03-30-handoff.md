## Root cause: Z.AI vs Zhipu account confusion

The original integration treated Z.AI and Zhipu as the same account. They are different:

| | Zhipu | Z.AI Coding |
|---|---|---|
| Base URL | `open.bigmodel.cn/api/paas/v4` | `api.z.ai/api/coding/paas/v4` |
| API key env | `ZHIPU_API_KEY` | `Z_AI_API_KEY` |
| Model IDs | `glm-5-20250605`, `glm-4v`, ... | `glm-5.1`, `glm-5-turbo`, `glm-4.7-flashx` |
| Billing | Pay-per-token | Subscription (Coding Plan) |

The `/models` endpoint on `api.z.ai/api/coding/paas/v4` returns empty — dashboard "Add Custom Provider" cannot auto-discover models. Models must be specified by name in the catalog.

## Key decisions

| Decision | Rationale |
|----------|-----------|
| Remove standalone `zai` provider | User has no standard Z.AI subscription; the separate card in the dashboard could accidentally delete the shared `Z_AI_API_KEY` |
| Three models only | User confirmed these are the only models they use from the Devpack |
| Zero-cost pricing in catalog | Devpack is subscription-billed, not per-token; fake costs would skew usage stats |
| `glm` shortcut → `glm-5.1` | Old target `glm-5-20250605` was on Zhipu, not Z.AI |
| Display name "Z.AI Coding" | "Devpack" is Z.AI marketing; unnecessary in UI |
| Full release build over `release-fast` | User preference for final binary; `release-fast` profile exists for faster iteration |
