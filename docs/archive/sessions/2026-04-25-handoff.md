  - **Before:** `std::mem::drop(self.memory.remove_agent(agent_id))` — drops async future

  - **After:**
    ```rust
    let _ = tokio::task::block_in_place(|| {
        tokio::runtime::Handle::current()
            .block_on(self.memory.remove_agent(agent_id))
    });
    ```
  - Same pattern as `spawn_agent` directly above it in the file
- **Discovered `memory-candle` feature flag** — Candle backends (embedding, NER, reranker,

flag; all four backends showed `active: false`.
- **Rebuilt and deployed** with `memory-candle`:
  ```bash
  cargo build --release --features memory-candle -p openfang-cli

  ```

## Discovered

- **File sandbox for wizard-created agents** — wizard agents resolve `file_read` relative
to workspace, not `~/.openfang/`. System prompts referencing `~/.openfang/config.toml`
etc. fail silently. Solution: API-first discovery, API key stored in `TOOLS.md`.
- `**[web.searxng]` only has `url` field** — `max_results` key in that section is silently
ignored. Config.toml had this stale key; noted in context.md gotchas.
- **Canon directory expanded** — rtr-openfang skill and OpenFangOps context.md reference
two canon files; there are now four. Both need updating.
- **Bundled agents are compile-time embedded** — `bundled_agents.rs` uses `include_str!`
for 30 templates. They install to `~/.openfang/agents/` on first run but are NOT
auto-spawned. Only `assistant` auto-spawns on empty registry.
- **OpenFangOps auto-registers from agent.toml** — after DB wipe, OpenFangOps re-appeared
without manual spawn. Mechanism not fully traced (not the bundled agent scan path).
Likely related to workspace presence or some post-boot directory scan not found in the
code read so far.
