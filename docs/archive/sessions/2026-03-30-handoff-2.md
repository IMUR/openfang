## Root cause analysis

The TUI's data fetching layer (`event.rs`) had two systemic problems:

1. **API evolution drift** — The API response shapes changed over time (field renames, envelope wrapping) but the TUI's parsing code was never updated. Every `spawn_fetch_*` function used `if let Ok(...)` chains that silently fell through to "no data" on parse failures.

2. **Shared mutable loading state** — A single `loading: bool` was shared across three Settings sub-tabs. Provider fetch completing could clear the loading flag while models were still fetching, creating a race where models appeared as "No models available" instead of showing a spinner.

Both problems compound: API mismatch causes silent parse failure → no success event sent → loading flag never cleared (stuck spinner) OR cleared by another tab's success (false "no data").

## Key decisions

| Decision | Rationale |
|----------|-----------|
| Split into two commits (fix + feature) | Fetcher alignment is a bug fix; slash commands are a new feature. Independent revertability. |
| `LoadState` enum over `Option<Result>` | Explicit state machine with `Idle`/`Loading`/`Loaded`/`Error` is self-documenting and renderable |
| Per-sub-tab state over global error | Providers, models, and tools fetch independently — one can fail while others succeed |
| Error message includes cause | "Connection failed: ..." vs "Invalid response: ..." tells the user whether it's a daemon issue or an API mismatch |
| Build with `--bin openfang` | Headless Debian system — full workspace build fails on `openfang-desktop` (GTK/gdk-sys dependency) |

- **Daemon stdout** — Previous daemon ran with stdout → `/dev/null`. Current daemon logs to `/tmp/openfang-daemon.log`. Consider a persistent log strategy (systemd journal, or `~/.openfang/daemon.log`).
- **Other tabs** — The `LoadState` pattern was only applied to Settings sub-tabs. The same silent-failure pattern exists in every other `spawn_fetch_*` function (dashboard, channels, sessions, etc.). The existing `FetchError(String)` event variant handles some tabs but routes based on active tab, which is fragile.
