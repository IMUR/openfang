- **Multi-destination cron delivery** — cron jobs fan out to channels
  (40 adapters), webhooks (optional auth), `LocalFile` (append/overwrite),

  `GET /api/commands?surface=web|cli|channel|all`. Prevents command drift
  across CLI, web chat, and 40 channels.
- **Atomic `config.toml` writer** — tempfile + rename prevents corruption
  on crash.
- **Kernel `skill_config_overrides`** — live config updates without
  restart.
- Clippy cleanups (`manual_contains`), desktop `.dmg`/`.msi` version
  match.

## Voice pipeline refactor assessment

User asked whether the voice pipeline consolidation warrants a full
refactor. **Recommendation: defer until after v0.6.0 merge.**

### Current shape

| File | LOC | Concerns in one file |
|---|---:|---|
| `crates/openfang-api/src/voice.rs` | 1506 | protocol / opus / session state / STT client / TTS client / WAV / resampling / markdown-to-speakable / sentence buffer |
| `crates/openfang-api/src/ws.rs` | 1866 | chat WS handler + voice WS dispatch + binary frame routing |
| `crates/openfang-api/static/js/voice-client.js` | 312 | already well-scoped, single responsibility |
| `crates/openfang-api/static/js/pages/chat.js` | 1435 | Alpine dashboard (voice is now a thin delegate) |
| `crates/openfang-runtime/src/candle_vad.rs` | 253 | Silero inference — clean |

### Refactor-worthy observations

`voice.rs` is ~4 logical modules glued together:

1. **Protocol** (`parse_binary_frame`, `encode_binary_frame`, `OpusCodec`,
   PCM helpers) — pure codec/serde, ~230 LOC
2. **Session** (`VoiceSession`, `VoiceSessionState`, `VoiceAction`,
   energy VAD, state transitions) — ~300 LOC, the heart of the pipeline
3. **Clients** (`SttClient`, `TtsClient`) — HTTP wrappers, ~150 LOC
4. **Audio utilities** (`pcm_to_wav`, `parse_wav`, `resample`,
   `markdown_to_speakable`, `SentenceBuffer`) — ~450 LOC

Splitting into a `voice/` module directory would be natural:
```
crates/openfang-api/src/voice/
├── mod.rs           // re-exports
├── protocol.rs      // frame parsing + opus codec
├── session.rs       // VoiceSession, state machine, VAD
├── clients.rs       // SttClient, TtsClient
└── audio.rs         // WAV, resample, markdown, sentence buffer
```

`ws.rs` similarly could shed ~500 LOC into `ws/voice.rs` to separate the
voice dispatch from chat WS.

### Why defer

1. **Merge friction.** `voice.rs` and `ws.rs` don't exist in upstream at
   all (our code is pure addition). A file split now creates
   rename-in-one-side conflicts during the v0.6.0 merge for zero
   functional benefit.
2. **Zero behavioral delta.** Reorganizing 3600 LOC across four files
   doesn't reduce code, improve performance, or fix bugs — it's
   ergonomic only. Ergonomics can wait.
3. **Post-merge is the natural window.** After v0.6.0 lands, the codebase
   will have churned enough (command registry, skill templates) that a
   fresh eye on voice module boundaries will be more informed.

### Why not "never"

The 1506-line `voice.rs` is genuinely hard to navigate (nine distinct
concerns crammed in one file, tests at the bottom making line numbers
jump). New contributors coming to the voice pipeline pay a
comprehension tax every time. Worth a ~half-day refactor after v0.6.0.
