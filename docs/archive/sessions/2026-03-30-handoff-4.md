## Discovered issues

### Four independent tool gates
An agent's effective tool access passes through: capabilities list, exec_policy, taint labels, and SSRF. Each blocks independently. This isn't documented anywhere.

### Template vs runtime state
Changes to `~/.openfang/agents/*/agent.toml` don't affect running agents. The daemon restores agents from SurrealDB on boot. Must delete agent and restart daemon to pick up template changes.

### MAX_PENDING_PER_AGENT = 5
Hardcoded in `approval.rs:15`. When an LLM fires 6+ tool calls needing approval, the 6th is instantly auto-denied. User sees tools failing before they can approve the first card.

### Misleading quota error
`ws.rs:1230` catches any error containing "quota" and rewrites it as "Token quota exceeded. Try /compact or /new." This fires for both OpenFang's internal metering limits AND upstream provider rate limits. Impossible to distinguish the cause from the user's perspective.

## Blocked items

- **`web_fetch` to localhost** — needs SSRF allowlist for daemon's own `api_listen` AND taint declassification for self-API calls. Code change required. Plan exists.
- **`MAX_PENDING_PER_AGENT`** — needs to be configurable or raised. Code change required. Plan exists.
- **7 inline approval card files still uncommitted** — carried from handoff-3.
- **Global exec_policy** — no `[exec_policy]` in `config.toml`. Kernel default may be too restrictive for this single-user deployment. Needs investigation.
